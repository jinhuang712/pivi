# 系统架构（大纲）

## 架构目标

- 在无业务云后端前提下，提供可用的多人语音频道
- 将复杂度收敛到客户端与房主节点两类组件
- 保留后续增加轻量网络辅助能力的扩展点
- **极低负载与极简分发**：采用 Rust + Tauri 架构，保证电竞时资源占用极低，并最终打包为可“一键转发”的独立安装包（Windows .exe / macOS .dmg）。

## 组件划分

- 客户端 App（每个成员）
  - UI 与交互状态
  - 音频设备与采集播放
  - 房间消息与屏幕共享控制
  - 信令会话与媒体连接管理
- 房主节点 Host Runtime（频道创建者）
  - 房间生命周期管理
  - 成员鉴权与频道状态广播
  - 房间消息广播与共享状态同步
  - P2P 失败时的媒体中转

## 逻辑拓扑

```mermaid
flowchart LR
    A[Member A Client] --> H[Host Runtime]
    B[Member B Client] --> H
    C[Member C Client] --> H
    A -. P2P Candidate .- B
    B -. P2P Candidate .- C
    A -->|Fallback Relay| H
    H -->|Relay| B
```

## 数据流分层

- 控制面（Control Plane）
  - 房间入口解析：基于 16 位自描述邀请码，直接恢复房主当前最优 `IP:Port` 入口，不依赖中心映射服务。
  - 创建房间、加入房间、成员状态、权限控制、房间消息（文本）
  - 房主迁移信令：新房主拉起 Host Runtime -> 老房主广播新连接信息 -> 全体重连
- 媒体面（Media Plane）
  - 音频流与屏幕共享流的采集、编码、传输、解码、播放
  - 基于 WebRTC DataChannel 的图片传输（避免大图片阻塞控制面的信令通道）

## 运行模式

- 模式 A：纯 P2P 优先（性能好，主机压力小）
- 模式 B：房主中转兜底（连通性更稳，主机压力增大）
- 模式 C：混合模式（部分成员直连，部分成员中转）

## 可扩展点

- 引入轻量 NAT 辅助服务（仅连通）
- 扩展 IPv6 / 更长邀请码版本
- 增加多候选入口编码与优先级切换
- 引入主机迁移（Host Failover）
- 引入细粒度权限角色体系
