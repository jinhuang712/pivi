# 产品详细设计总览 (MVP)

本文档是 Pivi 的终端用户体验与界面交互的**主入口**。为了保证需求的清晰度，具体的页面设计与交互规则已拆分至独立的子 PRD 文档中。

## 1. 核心体验原则

- **电竞级极简**：采用类似 Discord 的深色主题，左侧成员树，右侧共享与消息区，避免多余的层级跳转。
- **纯本地无云约束**：所有的设置（昵称、黑名单）都基于本地存储和 UUID 识别，不强制要求注册账号。
- **细粒度音频控制**：将“本地独立听感控制”与“房主全局控制”严格分离。

## 3. 关键交互流程串联

### 3.1 “房主全局闭麦” vs “本地屏蔽某人”
音频控制是开黑软件的核心体验，需严格区分**本地控制**与**服务端（房主）控制**：

1. **本地屏蔽某人（Local Mute）**：
   - **入口**：在频道主页左侧列表中，鼠标悬浮某人点击 `🔇`。
   - **效果**：只在**自己本地**把该路远端音频流（Remote Audio Track）的音量设为 0。其他人依然能听到他说话。
2. **房主全局闭麦（Server Mute / Host Mute）**：
   - **入口**：房主在左侧列表中右键点击某人，选择“全局强制闭麦”。
   - **效果**：房主 Runtime 拒绝转发该用户的音频包，并向所有人广播“该用户已被房主闭麦”的信令状态。此时该成员自己界面上的麦克风图标也会显示为红色的被锁状态，**所有人**都听不到他。

### 3.2 房主移交（Host Migration）流程
因为是无云架构，“移交房主”涉及到底层的连接重置：
1. 原房主在“设置中心 -> 移交房主权限”列表中选择目标成员 A。
2. 信令通知成员 A 客户端：“请在本地启动 Host Runtime 进程”。
3. 成员 A 的新进程启动并在本地（或通过 UPnP）获得可用端口后，回传地址给原房主。
4. 原房主向房间内所有人广播：“准备断线，重连至新房主地址”。
5. 所有客户端断开与旧房主的 WebSocket 和 WebRTC 连接，并根据新地址发起重连。
6. 旧房主销毁本地的 Host Runtime 进程，降级为普通成员。

### 3.4 创建与加入房间流程 (6 位 Code 机制)
为了降低局域网/跨网连通的心智负担，房间 ID 采用类似 Apple 设备验证的 **6 位字母+数字的 Room Code**。

1. **创建房间**：
   - 用户输入**房间名称**（如“周末上分小队”）点击创建。
   - 客户端在本地拉起 Host Runtime，并自动生成并分配一个全局唯一的 6 位字母+数字 Code（如 `A9B2K8`）。
   - 该 Code 会自动注册到轻量级公网发现服务（用于匹配主机 IP/Port 与房间名）。
2. **加入房间（二次确认弹窗）**：
   - 成员在加入界面，出现 6 个独立字符输入框（支持自动焦点跳转与粘贴解析）。
   - 用户输入 `A9B2K8` 后，客户端向发现服务查询该 Code 对应的房间信息。
   - 屏幕弹出二次确认模态框：“**是否加入房间：周末上分小队？**”（展示房主昵称与当前在线人数）。
   - 用户点击“确认加入”后，正式发起 WebRTC/WebSocket 连接流程。

### 3.5 快速图片发送与拦截流程
1. 用户在聊天框聚焦时按下 `Ctrl+V`。
2. 前端监听到 `paste` 事件，读取剪贴板中的 `File` 对象。
3. **前置校验**：若 `file.size > 5 * 1024 * 1024`，立即阻断流程，触发输入框上方的红色错误气泡。
4. **传输机制**：校验通过后，不走 WebSocket 信令（防挤占），而是通过 WebRTC 的 `DataChannel` 发送给其他客户端。
5. 接收端收到完整二进制数据后，通过 `URL.createObjectURL(blob)` 渲染到聊天列表中。
