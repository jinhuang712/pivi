# 无云统一邀请码协议设计（Self-Describing Invite Code）

## 1. 设计目标

- 在**不依赖业务云后端 / 共享发现服务**的前提下，为所有加入场景提供**唯一一种邀请码交互**。
- 让加入端仅通过邀请码即可恢复出连接入口，不需要额外查询中心映射表。
- 优先提升：
  - 房主建房后的“邀请码可用性稳定度”
  - 成员拿到邀请码后的“连接速度与成功率”

## 2. 单一邀请码模式

- 全场景统一使用 **16 位自描述邀请码**。
- 推荐展示格式：
  - `K7M2-9Q4P-T8XD-3F6N`
- 推荐编码字母表：
  - **Crockford Base32**
- 用户体验目标：
  - 只有一种输入方式
  - 支持自动分组、自动跳格、整串粘贴
  - 支持一键复制文本邀请码
  - 无论同网或跨网，UI 都不暴露“LAN / WAN 模式切换”

## 3. 为什么只保留一种模式

- 多模式接入会把复杂度转嫁给用户：
  - 用户要理解“这是局域网码还是跨网码”
  - 用户需要根据网络环境决定输入方式
  - UI 和错误提示会出现双路径分叉
- 统一邀请码的策略是：
  - **由 Host Runtime 选择当前最优入口**
  - **由邀请码携带该入口信息**
  - **由 Joiner 无脑输入同一种邀请码**

## 4. 为什么不再保留 6 位码

- IPv4 地址需要 32 bit。
- 端口需要 16 bit。
- 合计最少 48 bit，还未包含协议版本、端点类型、校验、过期时间等字段。
- 6 位 Base36 约只有 31 bit，6 位 Base62 约只有 36 bit，均无法安全承载跨网连接信息。
- 统一邀请码若要兼容同网与跨网，必须承载真实入口与完整校验，因此推荐固定为 **16 位 Base32**。

## 5. 16 位邀请码二进制布局（MVP）

- 总长度：**80 bit**
- 编码后固定为 **16 位 Base32**

### 4.1 字段布局

- `version`：4 bit
  - 协议版本，MVP 固定为 `0001`
- `endpoint_scope`：2 bit
  - `00 = private-lan-ipv4`
  - `01 = public-mapped-ipv4`
  - `10 = public-direct-ipv4`
  - `11 = reserved`
- `join_mode`：2 bit
  - `00 = direct-host`
  - `01 = host-relay-preferred`
  - 其余值保留
- `ipv4`：32 bit
  - 当前最优入口对应的 IPv4 地址
- `port`：16 bit
  - 房主信令入口端口
- `expiry_slot`：10 bit
  - 以 5 分钟为粒度的过期时间槽
  - 解码端超时后直接拒绝连接
- `checksum`：14 bit
  - 对前 66 bit 做 CRC/截断校验
  - 用于输入错误快速识别

### 5.2 设计约束

- MVP 先仅支持 `IPv4 + 单入口端口`。
- 后续如需支持 IPv6，不沿用现有 16 位格式，而是升级版本号并切换为更长编码。
- `checksum` 只解决误输入与误粘贴，不提供加密能力。

## 6. 房主侧“最优入口选择”与稳定性提升

### 6.1 入口优先级

- Host Runtime 启动后按以下顺序选择邀请码内写入的入口：
  1. **公网映射入口**：`UPnP / NAT-PMP / PCP` 成功后的公网 `IPv4:Port`
  2. **公网直连入口**：房主设备本身已具备公网 IPv4 且端口可达
  3. **局域网入口**：本机私网 IPv4（仅适用于同网设备）
- 一旦更高优先级入口探测成功，应重新签发邀请码。

### 6.2 建房稳定性优化

- **先监听，再出码**：只有在 TCP/WS 监听成功后才允许 UI 展示邀请码。
- **先探测，再出码**：邀请码展示前必须完成至少一次端口可达性自检。
- **端口预热**：
  - 优先复用上次成功的监听端口
  - 维护固定的小端口池，减少随机端口导致的路由器映射波动
- **并行准备**：
  - 启动监听
  - 申请端口映射
  - 探测公网 IP
  - 本地网络接口筛选
  以上步骤应并行执行，缩短建房首屏等待
- **邀请码就绪状态机**：
  - `Booting`
  - `ListeningReady`
  - `ReachabilityChecking`
  - `InviteReady`
  - `InviteDegraded`
- **降级提示**：
  - 若只能生成局域网入口邀请码，UI 应明确提示“仅同网可加入”
  - 若公网入口已确认可达，则提示“跨网可加入”

### 6.3 成员入房速度与稳定性优化

- **本地纯解码**：拿到邀请码后无需请求任何发现服务，减少一跳网络延迟。
- **直连前预检**：
  - 先执行输入校验与 TTL 校验
  - 再执行 `tcp connect timeout` 短超时探测
- **快速重试梯度**：
  - 第 1 次连接：1 秒超时
  - 第 2 次连接：2 秒超时
  - 第 3 次连接：切换为“入口不可达”错误
- **连接建立前缓存**：
  - 提前创建 WebSocket / RTC 管理器
  - 预初始化音频设备与状态容器
- **Host 健康快照首包**：
  - 首个响应包携带房间名、在线人数、模式、主机负载等级
  - 减少额外查询次数

## 7. 编码与解码规则

### 7.1 房主侧编码流程

1. Host Runtime 启动并拿到可达入口：
   - 优先 `UPnP / NAT-PMP / PCP`
   - 失败则回退到局域网入口或手动端口映射
2. 采集：
   - `endpoint_scope`
   - `selected_ipv4`
   - `signaling_port`
   - `join_mode`
   - `expiry_slot`
3. 组装 80 bit payload。
4. 计算 `checksum`。
5. 以 Crockford Base32 编码。
6. 输出给 UI：
   - 原始 16 位邀请码
   - 分组展示字符串
   - 复制用纯文本邀请码

### 7.2 成员侧解码流程

1. 接收用户输入的邀请码文本，先执行：
   - 去掉 `-`
   - 大写归一化
   - 长度校验
   - 字符集校验
2. Base32 解码还原 payload。
3. 校验：
   - `version`
   - `checksum`
   - `expiry_slot`
4. 恢复 `ip:port`、`endpoint_scope` 与 `join_mode`。
5. 直接发起：
   - `ws://<ip>:<port>` 或 `wss://<ip>:<port>` 连接
6. 成功握手后继续 `JOIN_ROOM -> ROOM_STATE -> WebRTC 协商` 流程。

## 8. 分享形态

- 统一邀请码是唯一的分享形态。
- 当前阶段只支持：
  - 复制纯文本邀请码
  - 手动粘贴纯文本邀请码
- 当前阶段不引入：
  - 移动端配套交互
  - 二维码分享
  - 深链接跳转

## 9. 安全边界

- 自描述短码不是加密容器，拿到短码的人可以恢复出公网入口。
- 因此 Runtime 仍必须执行：
  - 房间口令 / join token 校验
  - 黑名单拦截
  - 频率限制
  - 连接来源的基本防护
- 推荐追加以下安全增强：
  - `room_nonce`
  - `host_public_key_fingerprint`
  - 更严格的短码 TTL

## 10. 失败回退策略

- 若短码解码失败：
  - 直接提示“邀请码无效或已过期”
- 若短码可解码但地址不可达：
  - 提示“房主入口不可达，请检查端口映射 / 防火墙”
- 若 WebRTC 直连失败：
  - 回退到房主中转模式
- 若房主仅具备局域网入口：
  - UI 明确显示“当前邀请码仅适用于同一网络”

## 11. MVP 落地范围

- 支持：
  - 单一 16 位邀请码
  - 局域网 / 公网统一输入体验
  - 房主公网入口检测与过期短码
  - 输入校验、校验码、过期拒绝
  - 建房监听就绪检查、端口映射预热、连接快速重试
- 暂不支持：
  - IPv6 短码
  - 零配置穿越所有复杂 NAT
  - 纯短码内嵌端到端身份认证
