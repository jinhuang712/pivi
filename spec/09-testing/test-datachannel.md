# DataChannel 数据传输与边界测试用例 (TDD)

## 1. 剪贴板与文件大小拦截

### [TC-DC-01] 合法大小图片粘贴发送
- **描述**：通过 `Ctrl+V` 粘贴小于 5MB 的图片，应成功触发分片发送逻辑。
- **测试步骤**：
  1. 模拟 `paste` 事件，注入一个大小为 2MB 的 File 对象（Image）。
- **预期结果**：
  - 前端不显示拦截报错。
  - DataChannel `send()` 方法被调用多次（发送 File Meta 及后续二进制 Chunks）。

### [TC-DC-02] 超限图片粘贴拦截
- **描述**：粘贴超过 5MB 的图片时，必须在前端被直接拦截。
- **测试步骤**：
  1. 模拟 `paste` 事件，注入一个大小为 6MB 的 File 对象。
- **预期结果**：
  - DataChannel `send()` 未被调用。
  - UI 触发红色气泡错误提示：“图片大小不能超过 5MB”。

## 2. 二进制分片 (Chunking) 算法与重组

### [TC-DC-03] 文件切片数量与边界计算验证
- **描述**：验证 Chunking 算法是否正确计算分片数。
- **测试步骤**：
  1. 输入一个大小为 `150,000` 字节的 Buffer。
  2. 设定 chunkSize = `64 * 1024` (65536 字节)。
- **预期结果**：
  - `totalChunks` 应等于 `Math.ceil(150000 / 65536) = 3`。
  - 最后一块 Chunk 的字节数应为 `18928`。

### [TC-DC-04] 接收端 ArrayBuffer 完整重组验证
- **描述**：验证接收端根据元数据正确组装 Blob。
- **前置条件**：模拟接收端，预先收到 `totalChunks: 3` 的 `FILE_META`。
- **测试步骤**：
  1. 依次触发 DataChannel `onmessage` 事件，传入 3 个二进制 ArrayBuffer。
- **预期结果**：
  - 在收到第 3 个 Chunk 时，触发完成回调。
  - 组装后的 Blob 大小等于原始文件大小。
  - 成功通过 `URL.createObjectURL(blob)` 生成可访问的本地 URL。

## 3. 并发与异常边界防爆

### [TC-DC-05] 接收端丢包超时内存防爆
- **描述**：如果网络原因导致某张图片的分片永远收不齐，接收端必须在超时后清理内存，防止 OOM。
- **测试步骤**：
  1. 接收端收到 `totalChunks: 10` 的元数据。
  2. 接收端收到 5 个 Chunk 后，网络中断（或发送方退出）。
  3. 等待超时时间（例如 30 秒）。
- **预期结果**：
  - 触发超时清理定时器。
  - 内存中该 `fileId` 对应的 Map/Array 引用被删除。

### [TC-DC-06] 发送端高并发阻塞限制
- **描述**：防止用户瞬间疯狂粘贴图片导致 DataChannel 缓冲区溢出。
- **测试步骤**：
  1. 设置发送队列最大并发任务数为 2。
  2. 连续快速执行 5 次 4MB 图片的发送动作。
- **预期结果**：
  - 只有前 2 个文件立即开始分片发送。
  - 后 3 个文件进入排队状态。
  - 当前 2 个文件中任意一个发送完成后，队列开始处理下一个文件。
